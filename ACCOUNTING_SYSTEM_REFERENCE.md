## 1. نظرة عامة على المعمارية

### 1.1 الفلسفة الأساسية

النظام المحاسبي في مسار **اختياري ومكمّل** — يُفعَّل عبر "وضع المحاسبة" (accounting mode toggle) المحفوظ في `localStorage`. العمليات المالية الأساسية (فواتير، مصروفات، مقبوضات) تعمل بشكل مستقل تماماً عن المحاسبة. عند تفعيل وضع المحاسبة، يبدأ النظام بتوليد قيود يومية تلقائية لكل عملية مالية.

**القاعدة الذهبية:** أخطاء المحاسبة لا تكسر العملية المالية الأصلية أبداً (Silent Failure Pattern).

## 2. دليل الحسابات الافتراضي

عند تفعيل وضع المحاسبة أو إنشاء منظمة جديدة، يُنشأ دليل حسابات افتراضي مصمم خصيصاً لشركات المقاولات السعودية. الدليل هرمي بـ 4 مستويات ومبني وفق المعايير السعودية.

### 2.1 شرح الأعمدة

| العمود | الوصف |
|--------|-------|
| **الكود** | رقم الحساب الفريد داخل المنظمة |
| **الاسم العربي** | اسم الحساب بالعربية |
| **English Name** | اسم الحساب بالإنجليزية |
| **النوع (Type)** | `ASSET` / `LIABILITY` / `EQUITY` / `REVENUE` / `EXPENSE` |
| **الطبيعة (Normal Balance)** | `DEBIT` (مدين) أو `CREDIT` (دائن) — الجانب الذي يزيد الحساب |
| **المستوى (Level)** | L1 (رئيسي) → L2 (فرعي) → L3 (تفصيلي) → L4 (أكثر تفصيلاً) |
| **isSystem** | حساب نظامي لا يمكن حذفه |
| **isPostable** | قابل للترحيل — يمكن تسجيل قيود عليه. الحسابات غير القابلة للترحيل هي حسابات تجميعية فقط |

### 3.1 الحسابات النظامية (SYSTEM_ACCOUNTS)

هذه الحسابات تُستخدم في `auto-journal.ts` لتوليد القيود التلقائية. يُشار إليها بالكود (code) وليس بالـ id:

```typescript
const SYSTEM_ACCOUNTS = {
  ACCOUNTS_RECEIVABLE:      "1120",  // العملاء — الذمم المدينة

  ACCOUNTS_PAYABLE:         "2110",  // الموردون — الذمم الدائنة

  SALARIES_PAYABLE:         "2140",  // رواتب وأجور مستحقة
  GOSI_PAYABLE:             "2170",  // تأمينات اجتماعية مستحقة

  SUBCONTRACTORS:           "5200",  // مقاولو باطن
```

| `REFUND` | 4300 | تُسجَّل كإيرادات أخرى (استرداد) |

### 3.3 فئات المصروفات المعفاة من ضريبة المدخلات (VAT-EXEMPT)

هذه الفئات **لا يُفصَل فيها مبلغ ضريبة المدخلات** (Input VAT) عند إنشاء القيد التلقائي — يُسجَّل المبلغ الكامل على حساب المصروف:

```
SALARIES, SALARY, GOVERNMENT_FEES, BANK_FEES, FINES, INSURANCE
```

**السبب:** هذه المصروفات إما معفاة من الضريبة أصلاً (رواتب)، أو لا تحتوي على ضريبة مدخلات قابلة للاسترداد (رسوم حكومية، غرامات).

بالنسبة لباقي الفئات، عند وجود VAT:
```
DR  حساب المصروف    (المبلغ بدون ضريبة)
DR  1150 ض.مدخلات   (مبلغ الضريبة)
    CR  البنك        (المبلغ الإجمالي)
```

## 4. القيود التلقائية — خريطة العمليات الـ 29

### 4.1 المبدأ العام

ملف `auto-journal.ts` (769 سطر) هو قلب النظام المحاسبي. يحتوي على دوال (handlers) تُستدعى من procedures المالية عند كل عملية. كل دالة:

1. تتحقق من تفعيل وضع المحاسبة (accounting mode)
2. تبحث عن حسابات النظام بالكود (code)
3. تُنشئ القيد مع حالة `POSTED` مباشرة (القيود التلقائية لا تمر بمرحلة `DRAFT`)
4. عند الخطأ: تسجّل في `orgAuditLog` ثم تتابع (silent failure)

### 4.2 دورة حياة القيد التلقائي

```
عملية مالية ──→ handler في auto-journal.ts
                     │
                     ├─ تحقق من accounting mode (إذا غير مفعّل → return)
                     ├─ جلب الحسابات بالكود
                     ├─ حساب المبالغ (مع فصل VAT إن وجد)
                     ├─ إنشاء JournalEntry بحالة POSTED
                     │     ├─ referenceType = نوع العملية
                     │     ├─ referenceId = ID العملية الأصلية
                     │     ├─ referenceNo = رقم العملية الأصلية
                     │     ├─ isAutoGenerated = true
                     │     └─ entryNo = PREFIX-YYYY-NNNN
                     └─ إنشاء JournalEntryLine[] (مدين + دائن)
```

### 4.4 آلية العكس (Reversal)

عند حذف أو إلغاء عملية مالية:

1. يبحث النظام عن القيد الأصلي بواسطة `referenceType` + `referenceId`
2. يُنشئ قيداً عكسياً (يعكس المدين والدائن)
3. يُعلّم القيد الأصلي بـ `isReversed = true` + `reversalId`
4. القيد العكسي يأخذ `referenceType = REVERSAL` و `entryNo` بـ prefix `REV-JE`
5. حالة القيد الأصلي تتغير إلى `REVERSED`

```
القيد الأصلي (INV-JE-2026-0001):
  DR 1120  115,000
  CR 4100  100,000
  CR 2130   15,000
  status: POSTED → REVERSED
  isReversed: true
  reversalId: → يشير للقيد العكسي

القيد العكسي (REV-JE-2026-0001):
  DR 4100  100,000
  DR 2130   15,000
  CR 1120  115,000
  status: POSTED
  referenceType: REVERSAL
```

### 5.2 خوارزمية التوليد

```
الصيغة: {PREFIX}-{YEAR}-{SEQUENCE}

1. يُحدَّد الـ PREFIX من الجدول أعلاه بناءً على referenceType
2. السنة (YEAR) = السنة الميلادية الحالية (4 أرقام)
3. الـ SEQUENCE = آخر رقم مستخدم + 1 (لكل منظمة + prefix + سنة)
4. الرقم يُملأ بأصفار لـ 4 خانات (0001, 0002, ..., 9999)
```

> **ملاحظة:** `ORG_PAYMENT` و `INVOICE_PAYMENT` يشتركان في نفس الـ prefix (`RCV-JE`) لأنهما نوعان من المقبوضات. لكن الـ sequence منفصل لكل `referenceType`.

| فحص الصحة المحاسبية (Health Check) | 4 فحوصات للتحقق من سلامة البيانات | `/health-check` |

> **ملاحظة مهمة:** يوجد قائمتا دخل — واحدة مبنية على العمليات المالية المباشرة (Income Statement) والأخرى مبنية على القيود المحاسبية (Journal Income Statement). في النظام المثالي يجب أن تتطابقا، لكنهما قد تختلفان بسبب القيود اليدوية أو الأرصدة الافتتاحية أو العمليات المفقودة.

## 7. فحص الصحة المحاسبية (Health Check)

### 7.1 الفحوصات الأربعة

صفحة `/finance/accounting-reports/health-check` (مبنية على `accounting-health.ts` — 87 سطر) تُجري 4 فحوصات:

#### الفحص 1: قيود غير متوازنة (Unbalanced Entries)

```
الشرط:  القيود بحالة POSTED حيث SUM(debit) != SUM(credit)
التسامح: 0.01 ر.س (لتجنب أخطاء التقريب العشري)
الخطورة: حرجة — تعني خللاً في منطق إنشاء القيد
```

| المعلومات المعروضة | التفاصيل |
|--------------------|----------|
| رقم القيد | `entryNo` |
| مجموع المدين | `SUM(debit)` |
| مجموع الدائن | `SUM(credit)` |
| الفرق | `ABS(SUM(debit) - SUM(credit))` |

#### الفحص 2: فواتير بدون قيود (Invoices Without Entries)

```
الشرط:  فواتير بحالة ISSUED أو PAID ليس لها قيد من نوع INVOICE
الخطورة: عالية — تعني أن الإيراد غير مسجل محاسبياً
الحل:   تشغيل الترحيل الاسترجاعي (backfill)
```

#### الفحص 3: قيود يتيمة (Orphaned Entries)

```
الشرط:  قيود من نوع INVOICE لكن الفاتورة المرجعية محذوفة
الخطورة: متوسطة — قيود تشير لعمليات غير موجودة
الحل:   عكس القيود اليتيمة يدوياً
```

#### الفحص 4: مصروفات بدون قيود (Expenses Without Entries)

```
الشرط:  مصروفات بحالة COMPLETED ليس لها قيد من نوع EXPENSE
الخطورة: عالية — تعني أن المصروف غير مسجل محاسبياً
الحل:   تشغيل الترحيل الاسترجاعي (backfill)
```

## 8. الترحيل الاسترجاعي (Backfill)

### 8.1 الغرض

عند تفعيل وضع المحاسبة لأول مرة على منظمة لديها عمليات مالية سابقة، يحتاج النظام لإنشاء قيود تلقائية لكل هذه العمليات. هذه العملية تُسمى "الترحيل الاسترجاعي" (Backfill).

**ملف:** `packages/api/lib/accounting/backfill.ts` (246 سطر)

### 8.2 الأنواع الـ 11 المشمولة

يُنفَّذ الترحيل الاسترجاعي بالترتيب التالي (الترتيب مهم لضمان سلامة المراجع):

| # | النوع | الشرط | القيد المُولَّد | Reference Type |
|---|-------|-------|----------------|----------------|
| 1 | الفواتير | `status = ISSUED \| PAID` وليس لها قيد `INVOICE` | DR 1120 / CR 4100 + 2130 | `INVOICE` |
| 2 | الإشعارات الدائنة | `type = CREDIT_NOTE` وليس لها قيد `CREDIT_NOTE` | DR 4100+2130 / CR 1120 | `CREDIT_NOTE` |
| 3 | دفعات الفواتير | `InvoicePayment` بدون قيد `INVOICE_PAYMENT` | DR بنك / CR 1120 | `INVOICE_PAYMENT` |
| 4 | المصروفات المكتملة | `status = COMPLETED` وليس لها قيد `EXPENSE` | DR مصروف+VAT / CR بنك | `EXPENSE` |
| 5 | التحويلات المكتملة | `status = COMPLETED` وليس لها قيد `TRANSFER` | DR بنك2 / CR بنك1 | `TRANSFER` |
| 6 | دفعات مقاولي الباطن | `SubcontractPayment` بدون قيد `SUBCONTRACT_PAYMENT` | DR 5200\|2120 / CR بنك | `SUBCONTRACT_PAYMENT` |
| 7 | مستخلصات باطن معتمدة | `status = APPROVED` بدون قيد `SUBCONTRACT_CLAIM_APPROVED` | DR 5200 / CR 2120 | `SUBCONTRACT_CLAIM_APPROVED` |
| 8 | مسيّرات رواتب معتمدة | `status = APPROVED \| PAID` بدون قيد `PAYROLL` | DR 6100 / CR بنك + 2170 | `PAYROLL` |
| 9 | المقبوضات المباشرة | `FinancePayment (type = INCOME)` بدون قيد `ORG_PAYMENT` | DR بنك / CR 4300 | `ORG_PAYMENT` |
| 10 | دفعات المشاريع | `ProjectPayment` بدون قيد `PROJECT_PAYMENT` | DR بنك / CR 1120 | `PROJECT_PAYMENT` |
| 11 | مستخلصات مشاريع معتمدة | `status = APPROVED` بدون قيد `PROJECT_CLAIM_APPROVED` | DR 1120 / CR 4100 | `PROJECT_CLAIM_APPROVED` |

### 8.3 آلية العمل

```
backfill(organizationId, prisma)
    │
    ├─ 1. جلب كل العمليات المالية للمنظمة
    ├─ 2. لكل نوع من الـ 11:
    │     ├─ فلترة العمليات التي ليس لها قيد (LEFT JOIN → NULL)
    │     ├─ لكل عملية مفقودة:
    │     │     ├─ استدعاء handler المناسب من auto-journal.ts
    │     │     └─ تسجيل النتيجة (نجاح / فشل)
    │     └─ تجميع الإحصائيات
    ├─ 3. إرجاع ملخص: { created: N, failed: M, details: [...] }
    └─ 4. عرض النتيجة في لوحة المحاسبة كتنبيه مؤقت
```

### 8.4 الاستدعاء

يُستدعى الترحيل الاسترجاعي من:
- **لوحة المحاسبة** — زر "ترحيل استرجاعي" في الإجراءات السريعة
- **أول تفعيل لوضع المحاسبة** — اقتراح تلقائي للمستخدم
- **API endpoint** — `accounting.backfill` (subscriptionProcedure)

## 10. نماذج قاعدة البيانات (Schema Models)

### 10.1 ChartAccount — دليل الحسابات

```prisma
model ChartAccount {
  id              String            @id @default(cuid())
  organizationId  String
  code            String            // رقم الحساب (فريد لكل منظمة)
  nameAr          String            // الاسم بالعربية
  nameEn          String            // الاسم بالإنجليزية
  type            ChartAccountType  // ASSET | LIABILITY | EQUITY | REVENUE | EXPENSE
  normalBalance   NormalBalance     // DEBIT | CREDIT
  level           Int               // 1 (رئيسي) → 4 (أكثر تفصيلاً)
  parentId        String?           // الحساب الأب (هرمي)
  isSystem        Boolean           // نظامي — لا يمكن حذفه
  isActive        Boolean           // مفعّل — يظهر في القوائم
  isPostable      Boolean           // قابل للترحيل — يمكن تسجيل قيود عليه
  bankAccount     OrganizationBank? // علاقة 1-1 مع البنك (إن وجد)

  // Indexes
  @@unique([organizationId, code])  // الكود فريد لكل منظمة
  @@index([organizationId])
  @@index([organizationId, type])
  @@index([organizationId, parentId])
}

enum ChartAccountType {
  ASSET
  LIABILITY
  EQUITY
  REVENUE
  EXPENSE
}

enum NormalBalance {
  DEBIT
  CREDIT
}
```

**ملاحظات:**
- العلاقة `bankAccount` هي one-to-one — كل بنك (`OrganizationBank`) يرتبط بحساب واحد في دليل الحسابات
- الحسابات غير القابلة للترحيل (`isPostable = false`) هي حسابات تجميعية تُجمع تحتها حسابات فرعية
- الكود `code` فريد لكل منظمة وليس عالمياً — منظمتان مختلفتان قد تستخدمان نفس الأكواد

### 10.2 JournalEntry — القيد اليومي

```prisma
model JournalEntry {
  id                String              @id @default(cuid())
  organizationId    String
  entryNo           String              // رقم القيد (PREFIX-YYYY-NNNN)
  date              DateTime            // تاريخ القيد
  description       String              // وصف القيد
  referenceType     String?             // نوع المرجع (INVOICE, EXPENSE, etc.)
  referenceId       String?             // ID العملية المرجعية
  referenceNo       String?             // رقم العملية المرجعية (INV-2026-0042)
  isAutoGenerated   Boolean             // true = قيد تلقائي | false = قيد يدوي
  status            JournalEntryStatus  // DRAFT | POSTED | REVERSED
  totalAmount       Decimal             @db.Decimal(15, 2) // إجمالي القيد
  isReversed        Boolean             // هل تم عكس هذا القيد؟
  reversalId        String?             // ID القيد العكسي (إن وجد)
  createdById       String?             // المستخدم الذي أنشأ القيد
  postedById        String?             // المستخدم الذي رحّل القيد
  postedAt          DateTime?           // تاريخ الترحيل
  lines             JournalEntryLine[]  // بنود القيد

  // Indexes
  @@unique([organizationId, entryNo])
  @@index([organizationId])
  @@index([organizationId, status])
  @@index([organizationId, referenceType, referenceId])
  @@index([organizationId, date])
}

enum JournalEntryStatus {
  DRAFT     // مسودة — قابلة للتعديل والحذف
  POSTED    // مرحّلة — لا تُعدَّل، فقط تُعكس
  REVERSED  // معكوسة — تم عكسها بقيد عكسي
}
```

**ملاحظات:**
- القيود التلقائية (`isAutoGenerated = true`) تُنشأ بحالة `POSTED` مباشرة
- القيود اليدوية تُنشأ بحالة `DRAFT` وتحتاج ترحيلاً صريحاً
- القيد المُعكَس يبقى في قاعدة البيانات بحالة `REVERSED` ولا يُحذف
- `totalAmount` = مجموع الجانب المدين (أو الدائن — يجب أن يتساويا)

**ملاحظات:**

### 10.5 RecurringJournalTemplate — قالب قيد متكرر

## 12. القيود والمحدوديات المعروفة

### 12.1 محدوديات وظيفية

| المحدودية | التأثير | حلول بديلة |
|-----------|--------|------------|
| **لا يوجد دعم عملات متعددة** | كل المبالغ بالريال السعودي فقط | لا يوجد — مصمم للسوق السعودي فقط حالياً |
| **لا يوجد دعم فروع متعددة** | لا يمكن فصل الحسابات بين الفروع | استخدام مراكز التكلفة (مشاريع) كبديل جزئي |
| **لا يوجد تكامل مع ERP خارجي** | لا يمكن تصدير/استيراد قيود من أنظمة أخرى | تصدير Excel يدوي |
| **لا يوجد دعم ZATCA Phase 2** | لا يمكن إصدار فواتير إلكترونية معتمدة من هيئة الزكاة | مخطط للتطوير المستقبلي |
| **قائمتا دخل** | Income Statement (مالي) و Journal Income Statement (محاسبي) قد يظهران أرقاماً مختلفة | طبيعي — الأول من العمليات المباشرة والثاني من القيود. يجب أن يتطابقا مع اكتمال البيانات |

### 12.2 محدوديات تقنية

| المحدودية | التأثير | التفاصيل |
|-----------|--------|----------|
| **نظام أرصدة مزدوج** | رصيد البنك في `OrganizationBank.balance` قد يختلف عن مجموع القيود المحاسبية على حساب البنك | `OrganizationBank.balance` يُحدَّث مباشرة عند كل عملية مالية. مجموع القيود المحاسبية يعتمد على وجود قيود لكل عملية. إذا فُقد قيد (silent failure) يحدث تباين. التسوية البنكية تساعد في كشف الفرق |
| **الفشل الصامت (Silent Failure)** | بعض القيود التلقائية قد لا تُنشأ | عند فشل إنشاء قيد تلقائي، يُسجَّل الخطأ في `orgAuditLog` لكن العملية المالية تتم بنجاح. فحص الصحة (Health Check) يكشف هذه الحالات، والترحيل الاسترجاعي (Backfill) يعالجها |
| **ترتيب العمليات في الـ Backfill** | الترتيب مهم — الفواتير قبل الدفعات | إذا تم تغيير ترتيب الـ backfill قد تنشأ قيود بمراجع غير صحيحة |
| **Decimal vs Number** | Prisma يُرجع `Decimal` وليس `number` | يجب دائماً استخدام `Number()` للتحويل قبل العمليات الحسابية. مثال: `Number(entry.totalAmount)` |

```typescript
export async function onInvoiceIssued(
  prisma: PrismaClient,
  organizationId: string,
  invoice: {
    id: string;
    invoiceNo: string;
    total: Decimal;
    vatAmount: Decimal;
    subtotal: Decimal;
    projectId?: string;
  },
) {
  try {
    // 1. جلب الحسابات النظامية
    const arAccount = await findAccountByCode(prisma, organizationId, SYSTEM_ACCOUNTS.ACCOUNTS_RECEIVABLE);
    const revenueAccount = await findAccountByCode(prisma, organizationId, SYSTEM_ACCOUNTS.PROJECT_REVENUE);
    const vatAccount = await findAccountByCode(prisma, organizationId, SYSTEM_ACCOUNTS.VAT_PAYABLE);

    if (!arAccount || !revenueAccount || !vatAccount) return;

    // 2. إنشاء القيد
    await createJournalEntry(prisma, {
      organizationId,
      date: new Date(),
      description: `قيد آلي — فاتورة ${invoice.invoiceNo}`,
      referenceType: "INVOICE",
      referenceId: invoice.id,
      referenceNo: invoice.invoiceNo,
      isAutoGenerated: true,
      status: "POSTED",  // القيود التلقائية تُرحَّل مباشرة
      lines: [
        { accountId: arAccount.id, debit: Number(invoice.total), credit: 0 },
        { accountId: revenueAccount.id, debit: 0, credit: Number(invoice.subtotal) },
        { accountId: vatAccount.id, debit: 0, credit: Number(invoice.vatAmount) },
      ],
      projectId: invoice.projectId,
    });
  } catch (error) {
    // Silent failure — لا نكسر العملية المالية
    console.error("Failed to create invoice journal entry:", error);
    // يُسجَّل في orgAuditLog
  }
}
```

### 13.4 نمط عكس القيد

```typescript
export async function reverseAutoJournalEntry(
  prisma: PrismaClient,
  organizationId: string,
  referenceType: string,
  referenceId: string,
) {
  // 1. البحث عن القيد الأصلي
  const original = await prisma.journalEntry.findFirst({
    where: { organizationId, referenceType, referenceId, status: "POSTED" },
    include: { lines: true },
  });

  if (!original) return; // لا يوجد قيد للعكس

  // 2. إنشاء القيد العكسي (عكس المدين والدائن)
  const reversalEntry = await createJournalEntry(prisma, {
    organizationId,
    date: new Date(),
    description: `عكس قيد — ${original.entryNo}`,
    referenceType: "REVERSAL",
    referenceId: original.id,
    isAutoGenerated: true,
    status: "POSTED",
    lines: original.lines.map(line => ({
      accountId: line.accountId,
      debit: Number(line.credit),   // عكس: الدائن يصبح مدين
      credit: Number(line.debit),   // عكس: المدين يصبح دائن
      description: line.description,
      projectId: line.projectId,
    })),
  });

  // 3. تعليم القيد الأصلي كمعكوس
  await prisma.journalEntry.update({
    where: { id: original.id },
    data: {
      status: "REVERSED",
      isReversed: true,
      reversalId: reversalEntry.id,
    },
  });
}
```

### 13.7 قواعد حاكمة مهمة

| القاعدة | التفاصيل |
|---------|----------|
| **organizationId على كل استعلام** | لا استثناءات — Multi-tenancy يتطلب فلترة دائمة |
| **Decimal → Number()** | كل حقول `Decimal(15,2)` يجب تحويلها بـ `Number()` قبل الحساب |
| **Silent Failure** | أخطاء القيود التلقائية لا تمنع العملية المالية — تُسجَّل في audit log فقط |
| **القيود التلقائية = POSTED** | لا تمر بمرحلة DRAFT — تُرحَّل فوراً |
| **القيود اليدوية = DRAFT** | تحتاج ترحيلاً صريحاً من المستخدم |
| **العكس لا الحذف** | القيود المرحّلة لا تُحذف أبداً — تُعكس بقيد عكسي |
| **تسامح 0.01** | فرق التوازن أقل من 0.01 ر.س يُعتبر مقبولاً (أخطاء تقريب Decimal) |
| **الفترة المغلقة** | لا يمكن إنشاء أو ترحيل قيود بتاريخ يقع ضمن فترة مغلقة |
| **RTL** | استخدم `ms-`/`me-`/`ps-`/`pe-`/`text-start`/`text-end` — لا تستخدم `ml-`/`mr-`/`text-left` |
| **الترجمة** | كل نص يظهر للمستخدم يمر عبر `useTranslations()` — لا نصوص مكتوبة مباشرة |

---

> **نهاية المرجع** — هذا المستند يُحدَّث مع كل تطوير كبير في النظام المحاسبي.
