### 1.2 Bug #3: createJournalEntry لا يستخدم $transaction

**الملف:** `packages/database/prisma/queries/accounting.ts`

**التغيير في `createJournalEntry()` (~سطر 332-469):**

الـ Prisma nested create (`journalEntry.create` مع `lines: { create: [...] }`) يعمل كـ implicit transaction في Prisma — أي أن إنشاء الـ header والـ lines يتم في عملية واحدة ذرية.

**بعد المراجعة:** هذا **ليس خطأ فعلي** — Prisma's nested create هو ذري بطبيعته. لكن المشكلة أن العمليات **قبل** الـ create (duplicate check, period check, account validation) ليست في نفس الـ transaction. الحل:

```typescript
export async function createJournalEntry(db: PrismaClient, data: { ... }) {
  // Validations (lines, balance) — keep outside transaction (pure computation)
  if (data.lines.length < 2) throw ...;
  if (!totalDebit.equals(totalCredit)) throw ...;

  // Wrap DB operations in transaction
  return db.$transaction(async (tx) => {
    // 1. Duplicate check
    if (data.isAutoGenerated && data.referenceType && data.referenceId) {
      const existing = await tx.journalEntry.findFirst({ ... });
      if (existing) return existing;
    }

    // 2. Period check
    const periodClosed = await isPeriodClosedTx(tx, data.organizationId, data.date);
    // ... handle closed period

    // 3. Account validation
    const accounts = await tx.chartAccount.findMany({ ... });
    // ... validate

    // 4. Generate entryNo (outside tx — uses raw SQL which is its own tx)
    const seqValue = await nextSequenceValue(data.organizationId, `${prefix}-${year}`);
    const entryNo = formatSequenceNo(prefix, year, seqValue);

    // 5. Create entry + lines (atomic)
    return tx.journalEntry.create({ ... });
  });
}
```

**ملاحظة:** يجب إنشاء نسخة `isPeriodClosedTx` تقبل `tx` بدلاً من `db`.

### 2.4 Bug #2: لا يوجد Unique Constraint على referenceType + referenceId

**الملف:** `packages/database/prisma/schema.prisma`

**التغيير:** في model `JournalEntry`، غيّر الـ index إلى unique:

```prisma
// قبل:
@@index([organizationId, referenceType, referenceId])

// بعد:
// ⚠️ لا يمكن استخدام @@unique مباشرة لأن referenceType و referenceId قد يكونان null
// الحل: أبقِ الـ index + أضف validation في createJournalEntry (الموجود بالفعل في سطر 356-366)
```

**في الواقع:** بعد مراجعة الكود، `createJournalEntry` بالفعل يتحقق من التكرار (سطر 356-366). المشكلة أن هذا التحقق ليس ذري مع الـ create. الحل الأفضل هو لف العملية في `$transaction` (Bug #3 أعلاه يحل هذا).

**إضافة unique constraint مشروطة** (composite unique مع شرط أن الحقول ليست null):

بما أن Prisma لا يدعم conditional unique، الحل البديل هو إنشاء partial unique index عبر migration:

```sql
CREATE UNIQUE INDEX journal_entry_reference_unique
ON journal_entries (organization_id, reference_type, reference_id)
WHERE reference_type IS NOT NULL
  AND reference_id IS NOT NULL
  AND status != 'REVERSED';
```

**ملف Migration:** أنشئ ملف migration يدوي:
```
packages/database/prisma/migrations/YYYYMMDD_add_journal_reference_unique/migration.sql
```
