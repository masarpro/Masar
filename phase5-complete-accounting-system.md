# إكمال النظام المحاسبي — برومبت Plan Mode لـ Claude Code

## السياق — اقرأ هذا أولاً بالكامل

مسار (Masar) منصة SaaS لإدارة مشاريع المقاولات. تم بناء نظام محاسبي يتكون من:

### ما تم تنفيذه فعلاً (مباشر على الإنتاج app-masar.com):
1. **دليل حسابات (Chart of Accounts):** شجرة هرمية 1000-6000 (أصول، خصوم، حقوق ملكية، إيرادات، تكاليف مشاريع، مصروفات تشغيلية) مع حسابات نظام + بحث + Sidebar navigation
2. **قيود يومية تلقائية (Auto Journal Entries):** تتولّد عند إصدار فاتورة — مثلاً: مدين: العملاء 1120 / دائن: إيرادات 4100 + ضريبة 2130. ترقيم JE-2026-XXXX. حالات: مسودة/مرحّل/معكوس
3. **فترات محاسبية (Accounting Periods):** 12 فترة شهرية لكل سنة، مع إقفال وإعادة فتح
4. **صفحة التقارير المحاسبية:** 7 بطاقات (مديونيات متقادمة، مستحقات متقادمة، VAT، قائمة دخل مبسطة، قائمة دخل تفصيلية، ميزان مراجعة، ميزانية عمومية)
5. **Toggle "وضع المحاسبة":** يظهر/يخفي كل الأقسام المحاسبية
6. **تفاصيل القيد:** عرض بنود القيد مع رمز الحساب واسمه والمبلغ المدين/الدائن والإجمالي

### الـ Models المحاسبية الموجودة في Schema:
- `Account` — دليل الحسابات (code, nameAr, nameEn, type, normalBalance, level, parentId, isSystem, isPostable, bankAccountId)
- `JournalEntry` — القيد اليومي (entryNo, date, description, referenceType, referenceId, referenceNo, isAutoGenerated, status, totalAmount, adjustmentType, periodId)
- `JournalEntryLine` — بند القيد (accountId, debit, credit, description, projectId)
- `AccountingPeriod` — الفترة المحاسبية (name, periodType, startDate, endDate, isClosed, closedById, closingEntryId)

### الـ Enums المحاسبية:
- `AccountType`: ASSET, LIABILITY, EQUITY, REVENUE, EXPENSE
- `NormalBalance`: DEBIT, CREDIT
- `JournalEntryStatus`: DRAFT, POSTED, REVERSED

### 🔴 المجموعة 1 — حرجة (الأولوية القصوى)

#### 1. دفتر الأستاذ / كشف حساب (Account Ledger)

المحاسب يضغط على أي حساب في دليل الحسابات ويرى كل الحركات عليه بالتسلسل الزمني مع الرصيد التراكمي بعد كل حركة.

**المطلوب:**
- Procedure جديد: `accounting.accounts.getLedger(accountId, dateFrom, dateTo, page, pageSize)`
- الاستعلام: كل `JournalEntryLine` المرتبطة بهذا الحساب، مع join على `JournalEntry` (للتاريخ والوصف والرقم والحالة)
- فقط القيود المرحّلة (status = POSTED)
- ترتيب: حسب التاريخ تصاعدياً ثم entryNo
- **الرصيد التراكمي (Running Balance):** يُحسب من القيد الأول حتى الحالي. إذا الحساب normalBalance = DEBIT: الرصيد = مجموع المدين - مجموع الدائن. إذا CREDIT: العكس.
- **الرصيد الافتتاحي:** مجموع كل الحركات قبل dateFrom
- صفحة UI: `/finance/chart-of-accounts/[accountId]/ledger` أو dialog/drawer من صفحة دليل الحسابات
- الجدول: التاريخ | رقم القيد (رابط) | الوصف | المرجع | مدين | دائن | الرصيد
- في دليل الحسابات: كل حساب يصير قابل للنقر ← يفتح دفتر الأستاذ

#### 2. الأرصدة الافتتاحية (Opening Balances)

عندما يبدأ المحاسب باستخدام مسار، شركته لديها أرصدة حالية في البنوك والعملاء والموردين.

**المطلوب:**
- صفحة مخصصة: `/finance/opening-balances`
- تعرض كل الحسابات `isPostable = true` مع حقل إدخال مبلغ لكل حساب (مدين أو دائن)
- عند الحفظ: يُنشئ قيد افتتاحي واحد بتاريخ بداية السنة المالية
  - referenceType = "OPENING_BALANCE"
  - الطرف المقابل: حساب جديد "3000 — رأس المال" أو حساب موازنة خاص (ممكن يكون الأرباح المبقاة 3200)
- التحقق: مجموع المدين = مجموع الدائن
- يجب أن تُنفّذ مرة واحدة فقط — بعدها تُعدّل عبر تعديل القيد الافتتاحي
- Procedure: `accounting.openingBalances.get()` + `accounting.openingBalances.save(lines[])`
- إذا سبق وجود قيد افتتاحي: يُعدّل عليه بدل إنشاء جديد

#### 3. كشف حساب عميل / مقاول باطن (Client/Vendor Statement)

تقرير يُرسل للعميل للمطابقة أو لمقاول الباطن.

**المطلوب:**
- Procedure: `accounting.statements.clientStatement(clientId, dateFrom, dateTo)`
- مصدر البيانات: `FinanceInvoice` + `FinanceInvoicePayment` (لا يعتمد على القيود — يعمل حتى بدون المحاسبة)
- الهيكل:
  ```
  بيانات العميل (اسم، رقم ضريبي، عنوان)
  ───────────────────────────────────────────
  التاريخ | المرجع      | الوصف          | مدين   | دائن   | الرصيد
  01/01   | INV-0014    | فاتورة         | 11,500 |        | 11,500
  01/15   | PAY-0003    | دفعة           |        | 5,000  | 6,500
  02/01   | INV-0015    | فاتورة         | 13,800 |        | 20,300
  ───────────────────────────────────────────
  الرصيد المستحق: 20,300 ريال
  ```
- نفس الشيء لمقاولي الباطن: `accounting.statements.vendorStatement(contractId, dateFrom, dateTo)`
  - مصدر: `SubcontractClaim` + `SubcontractPayment`
- صفحة UI: زر "كشف حساب" في صفحة تفاصيل العميل + صفحة تفاصيل مقاول الباطن
- يجب أن يكون قابل للطباعة (CSS print styles) وتصدير PDF

#### 4. سند قبض وسند صرف (Receipt/Payment Voucher)

مستند رسمي مطبوع لكل عملية قبض أو صرف.

**المطلوب:**
- **سند القبض (Receipt Voucher - RV):** يُنشأ عند تسجيل دفعة على فاتورة أو مقبوض مباشر
  - رقم تسلسلي: `RV-2026-XXXX` (عبر OrganizationSequence)
  - الحقول: رقم السند، التاريخ، استلمنا من (اسم العميل)، المبلغ (رقم)، المبلغ بالحروف العربية، وذلك عن (الوصف)، طريقة الدفع، الحساب البنكي، رقم الفاتورة المرتبطة، رقم القيد المحاسبي، التوقيع
- **سند الصرف (Payment Voucher - PV):** يُنشأ عند دفع مصروف أو دفعة لمقاول باطن
  - رقم تسلسلي: `PV-2026-XXXX`
  - نفس الحقول مع "صُرف إلى" بدل "استلمنا من"
- **لا يحتاج model جديد في Schema** — يُولّد كمستند PDF/طباعة من البيانات الموجودة
- Procedure: `accounting.vouchers.getReceiptVoucher(paymentId, type: 'invoice_payment' | 'org_payment')`
- Procedure: `accounting.vouchers.getPaymentVoucher(expenseId | subcontractPaymentId)`
- الترقيم: يُخزّن في الـ metadata الموجود أو يُضاف حقل `voucherNo` اختياري على `FinanceInvoicePayment`, `FinancePayment`, `FinanceExpense`, `SubcontractPayment`
- **صفحة UI:** زر "طباعة سند" في كل مكان يتم فيه تسجيل دفعة (صفحة الفاتورة، المصروفات، المقبوضات، مدفوعات الباطن)
- **المبلغ بالحروف العربية:** دالة `numberToArabicWords(amount)` — مثال: `13,800` → `"ثلاثة عشر ألفاً وثمانمائة ريال سعودي"`
- القالب: يستخدم نظام قوالب مسار الحالي (FinanceTemplate) أو يكون قالب ثابت مُنسّق

#### 5. ترحيل جماعي للقيود (Bulk Post)

حالياً يجب فتح كل قيد على حدة لترحيله.

**المطلوب:**
- في صفحة القيود اليومية `/finance/journal-entries`:
  - Checkboxes على كل صف
  - "تحديد الكل" في Header
  - زر "ترحيل المحدد" يظهر عند تحديد قيد واحد أو أكثر
- Procedure: `accounting.journal.bulkPost(entryIds: string[])`
  - يتحقق أن كل القيود بحالة DRAFT
  - يتحقق أن كل القيود متوازنة
  - يتحقق أن لا يوجد قيود في فترات مغلقة
  - يُرحّل الكل في transaction واحد
  - يُرجع: عدد المُرحّلة + أي أخطاء
- زر "ترحيل كل المسودات" (بدون تحديد) — يُرحّل كل القيود بحالة DRAFT

### 🟠 المجموعة 2 — مهمة

#### 6. تقرير مراكز التكلفة حسب المشروع (Project Cost Center Report)

حقل `projectId` موجود فعلاً في `JournalEntryLine` لكن لا يوجد تقرير يستغله.

**المطلوب:**
- Procedure: `accounting.reports.costCenterByProject(organizationId, dateFrom, dateTo, projectId?)`
- يعرض: لكل مشروع → كل حسابات المصروفات والإيرادات المرتبطة → المبلغ
- هيكل:
  ```
  المشروع: فيلا الأمير — مشروع A
  ├── إيرادات المشاريع (4100): 500,000
  ├── مواد ومشتريات (5100): 120,000
  ├── مقاولو باطن (5200): 150,000
  ├── عمالة مباشرة (5300): 40,000
  └── صافي الربح: 190,000 (38%)
  ```
- صفحة UI: ضمن التقارير المحاسبية أو كصفحة منفصلة
- مع Recharts: مخطط مقارنة أرباح المشاريع

#### 7. تسوية بنكية محاسبية (Bank Reconciliation)

مسار عنده `reconcile` بسيط. المحاسب يحتاج تسوية حقيقية.

**المطلوب:**
- لا يحتاج Schema change كبير — ممكن إضافة model بسيط:
  ```
  BankReconciliation { id, bankAccountId, reconciliationDate, bankStatementBalance, systemBalance, difference, status, notes, reconciledById, reconciledAt }
  BankReconciliationItem { id, reconciliationId, journalEntryLineId?, description, amount, isMatched, matchedAt }
  ```
- أو يكون أبسط: صفحة تعرض حركات البنك (من القيود على حساب البنك المحاسبي) مع حقل "الرصيد حسب كشف البنك" وتحسب الفرق
- صفحة UI: `/finance/banks/[bankId]/reconciliation`
- المرحلة الأولى (الآن): مطابقة يدوية — المحاسب يُدخل رصيد كشف البنك ويعلّم الحركات المطابقة
- المرحلة الثانية (لاحقاً): استيراد كشف بنك CSV/Excel

#### 8. قيود متكررة (Recurring Journal Entries)

المحاسب يحتاج قيود تتكرر كل شهر (إيجار، إهلاك، اشتراكات).

**المطلوب:**
- إضافة حقول على `JournalEntry` أو model جديد:
  ```
  RecurringJournalTemplate { id, organizationId, description, lines[], frequency: MONTHLY|QUARTERLY|ANNUAL, dayOfMonth, isActive, lastGeneratedDate, nextDueDate }
  ```
- Procedure: `accounting.recurring.create(template)`, `accounting.recurring.list()`, `accounting.recurring.generate()` — يولّد القيود المستحقة
- يمكن ربطه بـ `generateMonthly` الموجود في مصروفات الشركة كمرجع للنمط
- صفحة UI: tab أو قسم في صفحة القيود اليومية
- زر "توليد القيود المتكررة" يُنشئ كل القيود المستحقة بحالة DRAFT

#### 9. طباعة وتصدير PDF لكل التقارير

لا يوجد حالياً زر طباعة أو تصدير في أي صفحة محاسبية.

**المطلوب:**
- زر "طباعة" + زر "تصدير PDF" في كل صفحة تقرير محاسبي:
  - ميزان المراجعة
  - قائمة الدخل
  - الميزانية العمومية
  - دفتر الأستاذ
  - كشف حساب العميل/المورد
- استخدم أبسط طريقة: CSS `@media print` مع `window.print()` كحل أولي
- هيدر الطباعة: اسم الشركة + الشعار + عنوان التقرير + التاريخ
- أو استخدم نفس نظام قوالب مسار (FinanceTemplate) إذا كان أسهل

#### 10. ربط ثنائي الاتجاه (Bidirectional Navigation)

حالياً "INVOICE: INV-2026-0015" يظهر كنص عادي في تفاصيل القيد.

**المطلوب:**
- **من القيد للمصدر:** في صفحة تفاصيل القيد (`/finance/journal-entries/[id]`)، اجعل referenceNo رابط قابل للنقر يفتح الفاتورة/المصروف/المقبوض/دفعة الباطن الأصلية
- **من المصدر للقيد:** في صفحة تفاصيل الفاتورة/المصروف/إلخ، أضف badge أو رابط "القيد المحاسبي: JE-2026-0002" يفتح تفاصيل القيد
- المنطق: ابحث في `JournalEntry` حيث `referenceType = 'INVOICE'` و `referenceId = invoiceId`
- الملفات المتأثرة:
  - صفحة تفاصيل القيد (موجودة)
  - صفحة تفاصيل الفاتورة (`/finance/invoices/[id]`)
  - صفحة تفاصيل المصروف (`/finance/expenses/[id]`)
  - صفحة تفاصيل المقبوض (`/finance/payments/[id]`)
  - صفحة مدفوعات الباطن

#### 11. ترقيم القيود حسب النوع

حالياً كل القيود: `JE-2026-XXXX`. المحاسب يفضّل ترقيم واضح.

**المطلوب:**
- تغيير prefix الترقيم حسب `referenceType`:
  - INVOICE → `INV-JE-2026-XXXX`
  - INVOICE_PAYMENT → `PAY-JE-2026-XXXX`
  - EXPENSE → `EXP-JE-2026-XXXX`
  - TRANSFER → `TRF-JE-2026-XXXX`
  - SUBCONTRACT_PAYMENT → `SUB-JE-2026-XXXX`
  - PAYROLL → `SAL-JE-2026-XXXX`
  - OPENING_BALANCE → `OPN-JE-2026-XXXX`
  - ADJUSTMENT → `ADJ-JE-2026-XXXX`
  - PERIOD_CLOSING → `CLS-JE-2026-XXXX`
  - MANUAL (يدوي) → `MAN-JE-2026-XXXX`
- يحتاج: تعديل دالة توليد `entryNo` في `createJournalEntry` لتقبل نوع المرجع
- كل نوع له sequence مستقل في `OrganizationSequence`

### 🟡 المجموعة 3 — تحسينية

#### 12. Dashboard محاسبي

**المطلوب:**

#### 13. بحث متقدم في القيود

**المطلوب:**
- في صفحة القيود `/finance/journal-entries`، إضافة فلاتر:
  - بالمبلغ: من/إلى (totalAmount)
  - بالحساب: dropdown يعرض الحسابات — يفلتر القيود التي تحتوي بند على هذا الحساب

  - بالتاريخ: من/إلى (موجود فعلاً)

#### 14. Audit Trail بصري

**المطلوب:**
- في صفحة تفاصيل القيد: قسم "سجل التدقيق" يعرض:
  - من أنشأ القيد (createdById) + متى (createdAt)

  - إذا معكوس: من عكسه + متى + رابط للقيد العكسي
  - إذا تلقائي: badge "تلقائي 🤖" مع نوع المصدر
- الحقول موجودة فعلاً في Schema — يحتاج فقط عرضها في UI

#### 15. تصدير Excel

**المطلوب:**

  - ميزان المراجعة
  - قائمة القيود اليومية
  - دفتر الأستاذ

#### 16. المبلغ بالحروف العربية

**المطلوب:**
- دالة utility: `numberToArabicWords(amount: number): string`
- تدعم: أعداد حتى 999,999,999
- تُضيف "ريال سعودي" و "هللة" تلقائياً
- مثال: `13800.50` → `"ثلاثة عشر ألفاً وثمانمائة ريال سعودي وخمسون هللة"`
- تُستخدم في: سندات القبض/الصرف + الفواتير (إذا مطلوب)
- مكان الملف: `packages/utils/src/number-to-arabic-words.ts`

#### 17. تقرير VAT تفصيلي حسب الفاتورة

**المطلوب:**
- توسيع تقرير VAT الموجود (أو صفحة فرعية):
  - جدول يعرض كل فاتورة: رقم الفاتورة | التاريخ | اسم العميل | المبلغ الخاضع | مبلغ الضريبة | نوع الفاتورة
  - نفس الشيء للمصروفات: رقم المصروف | التاريخ | الوصف | المبلغ الخاضع | مبلغ الضريبة
  - قابل للتصدير كـ Excel/CSV — لتقديمه كمرفق مع إقرار ZATCA

## ⛔ القائمة الحمراء — ملفات لا تلمسها

- `structural-calculations.ts`
- `derivation-engine.ts`
- `packages/api/modules/quantities/**`
- لا تكسر أي عملية مالية موجودة (فواتير، مصروفات، مقبوضات)
- لا تغيّر سلوك Toggle "وضع المحاسبة" — كل شيء جديد يظهر فقط عند تفعيله
- لا تعدّل الـ procedures المالية الحالية إلا لإضافة الربط الثنائي (badge القيد المحاسبي)

## 📋 أنماط يجب اتباعها

1. **اقرأ كل الملفات المذكورة أعلاه قبل وضع الخطة** — لا تفترض هيكل أي ملف
2. كل query يبدأ بـ `organizationId` (multi-tenancy إلزامي)
3. حقول مالية: `Prisma.Decimal` فقط — `@db.Decimal(15, 2)`
4. تقارير: `protectedProcedure` مع `verifyOrganizationAccess("finance.view")`
5. عمليات كتابة: `subscriptionProcedure`
6. RTL-safe: `ms-`, `me-`, `ps-`, `pe-` بدل `ml-`, `mr-`
7. ترجمة: `ar.json` + `en.json` بالتوازي
8. لا مكونات أكبر من 400 سطر — قسّم
9. استخدم `Promise.all` للاستعلامات المتوازية
10. بعد أي schema change: `pnpm --filter database db:generate`
11. تحقق بعد كل مرحلة: `pnpm tsc --noEmit && pnpm build`
