# برومبت تحليل جاهزية مسار للنشر — Claude Code

أنت تعمل على مشروع **"مسار" (Masar)** — منصة SaaS لإدارة شركات المقاولات في السوق السعودي. المشروع monorepo مبني بـ Next.js + Prisma + ORPC + BetterAuth.

## المطلوب منك

حلّل المشروع بالكامل وأنتج ملف `DEPLOYMENT_GUIDE.md` شامل ومفصّل بالعربية يشرح كيف أنقل هذا المشروع من بيئة التطوير المحلية إلى الإنترنت (Production Deployment).

## خطوات التحليل المطلوبة

### 1. تحليل البنية التحتية
اقرأ وحلّل هذه الملفات بالتحديد:
- `package.json` (الجذر + كل حزمة)
- `apps/web/next.config.ts`
- `apps/web/middleware.ts`
- `packages/database/prisma/schema.prisma`
- `packages/auth/auth.ts`
- `packages/api/orpc/router.ts` و `procedures.ts`
- `config/index.ts` (Feature Flags)
- `.github/workflows/*.yml`
- `turbo.json`
- `vercel.json` (إذا وجد)
- `.env.local.example` أو أي ملف env مرجعي
- `biome.json`
- `tooling/typescript/base.json`
- `.gitignore`

### 2. تحليل جاهزية كل وحدة
افحص `packages/api/modules/*/router.ts` واكتشف:
- أي endpoints على `publicProcedure` يجب أن تكون `protectedProcedure`
- أي وحدات ناقصة CRUD operations
- أي مشاكل أمنية واضحة

### 3. تحليل المتغيرات البيئية
اقرأ الكود واستخرج **كل** متغير بيئي مستخدم (`process.env.*` و `env.*`):
- صنّفها: إلزامي / موصى / اختياري
- اشرح كل متغير: ماذا يفعل، من أين تحصل عليه، ما القيمة المتوقعة

### 4. تحليل قاعدة البيانات
من `schema.prisma`:
- عدد النماذج والعلاقات
- هل هناك migrations جاهزة أم نستخدم db push؟
- هل هناك فهارس (indexes) مطلوبة للأداء في الإنتاج؟
- هل هناك حقول تحتاج تعديل نوع (مثل Float → Decimal)؟

### 5. تحليل الأمان
افحص:
- Security headers في next.config.ts
- Rate limiting implementation
- CORS settings
- Session configuration (freshAge, expiry)
- File upload validation
- هل كل الـ API endpoints محمية بشكل صحيح؟

### 6. تحليل الأداء في Serverless
حلّل التحديات المتوقعة عند النشر على Vercel:
- Cold starts مع Prisma
- Connection pooling requirements
- Function timeouts
- Cron jobs requirements
- WebSocket/real-time limitations

### 7. تحليل الخدمات الخارجية
اكتشف كل الخدمات الخارجية المستخدمة في الكود:
- مزودو البريد الإلكتروني
- مزودو الدفع
- مزودو التخزين
- مزودو المصادقة (OAuth)
- Redis
- AI APIs
- أي خدمة أخرى

## شكل الملف النهائي `DEPLOYMENT_GUIDE.md`

أنشئ الملف بالهيكل التالي (بالعربية، مفصّل جداً):

```markdown
# دليل نشر منصة مسار — من التطوير إلى الإنتاج

## 1. ملخص تنفيذي
   - حالة المشروع الحالية (أرقام: عدد الصفحات، النماذج، الـ endpoints)
   - تقييم الجاهزية العام (جاهز / جاهز مع تحفظات / غير جاهز)
   - قائمة المشاكل الحرجة التي يجب إصلاحها قبل النشر

## 2. المشاكل والثغرات المكتشفة
   - مشاكل أمنية (مع اسم الملف ورقم السطر إن أمكن)
   - مشاكل أداء
   - وحدات غير مكتملة
   - لكل مشكلة: الوصف + الملف + الحل المقترح + الوقت المتوقع للإصلاح

## 3. المتطلبات — ماذا تحتاج قبل البدء
   - حسابات مطلوبة (Vercel, Supabase, Resend, etc.)
   - أدوات مطلوبة على جهازك
   - نطاق (Domain)
   - تكاليف تقديرية شهرية

## 4. المتغيرات البيئية الكاملة
   - جدول بكل متغير: الاسم، الوصف، إلزامي/اختياري، من أين تحصل عليه، مثال
   - ملف .env.production جاهز للنسخ (بقيم placeholder)

## 5. خطوات النشر التفصيلية
   ### 5.1 إعداد قاعدة البيانات (Supabase)
   ### 5.2 إعداد التخزين (S3/Supabase Storage)
   ### 5.3 إعداد البريد الإلكتروني
   ### 5.4 إعداد Redis (إذا مطلوب)
   ### 5.5 إعداد OAuth (إذا مطلوب)
   ### 5.6 إعداد الدفع (إذا مطلوب)
   ### 5.7 تشغيل Migrations على قاعدة الإنتاج
   ### 5.8 النشر على Vercel (خطوة بخطوة)
   ### 5.9 ربط النطاق وSSL
   ### 5.10 اختبار ما بعد النشر

   كل خطوة يجب أن تتضمن:
   - ماذا تفعل بالضبط (screenshots ليست ممكنة، لكن وصف دقيق)
   - أي أمر terminal تحتاج تشغّله
   - أي ملف تحتاج تعدّله
   - ما النتيجة المتوقعة
   - ما الأخطاء المحتملة وكيف تحلّها

## 6. تحديات Vercel Serverless
   - جدول بكل تحدي والحل
   - إعدادات vercel.json المطلوبة
   - Cron jobs المطلوبة

## 7. قائمة مراجعة ما بعد النشر (Checklist)
   - اختبارات يدوية يجب تنفيذها
   - نقاط أمنية يجب التحقق منها
   - أداء يجب قياسه

## 8. حل المشاكل الشائعة (Troubleshooting)
   - لكل مشكلة محتملة: الأعراض + السبب + الحل

## 9. خطة الإطلاق المرحلية
   - أسبوع 1: إصلاحات + بنية تحتية
   - أسبوع 2-3: إطلاق تجريبي
   - أسبوع 4+: تحسينات
```

## تعليمات مهمة

1. **لا تفترض — اقرأ الكود الفعلي.** لا تكتب معلومات عامة عن Next.js أو Vercel. كل ما تكتبه يجب أن يكون مبنياً على تحليلك الفعلي لهذا المشروع بالذات.

2. **كن محدداً.** بدل "تأكد من إعداد المتغيرات البيئية" اكتب بالضبط أي متغيرات، وأين تجدها في الكود، وما القيم المطلوبة.

3. **اذكر أسماء الملفات الفعلية.** عندما تذكر مشكلة أو إعداد، اذكر المسار الكامل للملف.

4. **الملف بالعربية بالكامل** (المصطلحات التقنية بالإنجليزية مقبولة).

5. **احفظ الملف النهائي في جذر المشروع** باسم `DEPLOYMENT_GUIDE.md`.

6. **لا تعدّل أي كود** — فقط حلّل واكتب التقرير.

ابدأ الآن بتحليل المشروع وإنشاء الملف.
